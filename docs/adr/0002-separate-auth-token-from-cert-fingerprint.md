# ADR 0002: Decouple TLS identity from API authorization

## Status

Accepted

## Context

The initial security implementation used the SHA-256 fingerprint of the manager's self-signed Ed25519 certificate as both the cert-pinning value and the API authentication token. A single value served two distinct purposes:

1. Workers compared the fingerprint against the hash of the certificate sent during the TLS handshake to verify they were connecting to the legitimate manager.
2. The same value was sent in the `Netbula-Token` HTTP header by the control program to authenticate API requests.

But the design above is _flawed and insecure_ (given that the certificates are transmitted in plaintext during the initial TLS handshake)

- The fingerprint is directly derivable from the public certificate, any malicious actor on the network path can compute the SHA-256 hash and reconstruct the authentication token.

- This would allow an attacker to make unauthorized requests to the manager's API using the derived certificate fingerprint.

## Decision

We are decoupling the two roles into separate values with separate secrets:

- **Verifying the manager identity on worker/controller side (using Certificate Pinning):**
  - Workers and the control program are configured with the manager's `cert-fingerprint`: the SHA-256 hash of the manager's TLS certificate DER bytes.
  - During the TLS handshake, clients use `InsecureSkipVerify: true` combined with a `VerifyPeerCertificate` callback that rejects any certificate whose fingerprint does not match.
  - This prevents Man-in-the-Middle (MITM) attacks by ensuring clients only complete handshakes with the legit manager.

- **Verifying the controller identity on manager side (using Shared Secret key):**
  - A separate `auth-token` is generated by the manager at startup using `crypto/rand` (32 bytes, hex-encoded).
  - This token is then printed to the terminal for users to copy, this value needs to be supplied when we first start a Netbula control program.
  - The control program must supply it in the `Netbula-Token` HTTP header for all API requests. Because it is only transmitted inside an already-established TLS tunnel, it is never visible to passive observers.

Both workers and the control program can establish pinned TLS connections to the manager(Guaranteed to use the public key of the manager server).

## Consequences

- Pros:
  - No attacker can send commands to the manager without knowing the secret `auth-token` stored only on the `manager` server / machines that once run the `Netbula control` program
  - Allows `auth-token` rotation (e.g., after a suspected compromise) without regenerating the TLS certificate or redistributing the `cert-fingerprint` to all workers.
  - Separation of concerns: the fingerprint proves manager identity; the token proves controller authorization.

- Cons:
  - Slightly increases first-time configuration: users must collect and distribute two values from the manager's startup output instead of one.
  - Both values are stored at rest in plaintext config files (`worker_config.json`, `control_config.json`). Physical access to a node exposes its credentials.
  - The initial distribution of both values relies on a trusted out-of-band channel (e.g., SSH); there is no protection if the network is compromised before first setup.
